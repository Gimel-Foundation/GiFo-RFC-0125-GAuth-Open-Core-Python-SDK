## Abstract

This specification defines the Management API and Administration Interface for the Gimel Authorization (GAuth) protocol. While GiFo-RFC 0116 defines the credential format and schema for Power of Attorney (PoA) tokens, and GiFo-RFC 0117 defines how to enforce those credentials at runtime, no prior RFC specifies how to *create, query, modify, or revoke* PoA credentials through a standardized interface. This document fills that gap with two layers:

1. **Normative Core** — A Management API contract defining CRUD operations for mandate/PoA lifecycle management: creation with rule-based validation, querying and inspection, status transitions, suspension and resumption, revocation, budget management, TTL extension, delegation lifecycle, and governance profile assignment. All operations are deterministic and rule-based — no AI or probabilistic inference is required.

2. **Non-Normative Guidance** — Administration Interface patterns for management UIs: reference screen specifications for mandate creation, inspection, delegation management, budget monitoring, and governance profile assignment.

Together, these complete the open protocol triangle:

| RFC | Concern | Question Answered |
|-----|---------|-------------------|
| **RFC 0116** | Credential Format | What does a PoA look like? |
| **RFC 0117** | Enforcement | How do I check a PoA at runtime? |
| **RFC 0118** | Management (this document) | How do I create, query, modify, and revoke a PoA? |

This specification enables a fourth registration path — purely rule-based, deterministic, and AI-free — alongside the three AI-assisted registration paths that proprietary implementations (such as G-Agent 1) may provide. A third party implementing RFCs 0116, 0117, and 0118 can operate a complete GAuth-compatible authorization system without any proprietary dependencies.

## 1. INTRODUCTION

### 1.1 Purpose

GAuth defines a comprehensive authorization protocol for governing agent-based software systems. The protocol suite — RFCs 0110, 0111, 0115, 0116, and 0117 — specifies how PoA credentials are structured, issued, transported, validated, and enforced. However, it does not specify how those credentials are *managed* — i.e., how an administrator or automated system creates, inspects, modifies, and revokes mandates through a standardized interface.

Today, mandate lifecycle operations are handled by proprietary platform implementations. An AI-assisted system (such as G-Agent 1) may provide three registration paths with regulatory intelligence, risk scoring, and compliance reasoning. However, a third party building a GAuth-compatible system has no specification for the management operations — they can format a PoA (RFC 0116) and enforce it (RFC 0117), but cannot create or manage one through a published API contract.

This RFC provides that specification: a fourth registration path that is purely rule-based, deterministic, and AI-free. The caller provides all required mandate parameters; the Management API validates them against the PoA schema (RFC 0116) and governance profile ceilings (RFC 0115); and the system either accepts the mandate or returns structured validation errors.

### 1.2 Design Principles

1. **Rule-based** — All validation is deterministic and schema-driven. The Management API does not require AI, probabilistic inference, or regulatory interpretation to function. Given the same input and the same governance profile ceilings, the API MUST produce the same acceptance/rejection result.
2. **Schema-validated** — Every mandate submitted through the Management API is validated against the canonical PoA JSON Schema defined in RFC 0116 §4.3. Schema validation is the first gate; ceiling validation is the second.
3. **Ceiling-enforced** — Mandate parameters are validated against the governance profile ceiling table (RFC 0115). No mandate may exceed the ceilings defined by its declared governance profile. Ceiling violations produce structured errors, not silent truncation.
4. **Fail-closed** — If the Management API cannot complete a validation (missing ceiling data, schema resolution failure, internal error), the mandate is rejected. There is no auto-approve mode.
5. **Auditable** — Every management operation (create, update, suspend, resume, revoke, delegate) produces an audit record. The audit trail is immutable and includes the identity of the caller, the operation performed, the mandate affected, and the validation result.
6. **Extendable** — Proprietary features may layer additional registration paths with AI-assisted validation, risk scoring, regulatory interpretation, or implied-authority mechanisms on top of this API. Such extensions operate outside the normative contract defined in this specification and are excluded from the Apache 2.0 license scope (see Section 2). This is consistent with the Extendable principle defined in RFC 0117 §1.2.

### 1.3 Terminology

| Term | Definition |
|------|-----------|
| **Management API** | A standardized interface for creating, querying, modifying, and revoking PoA credentials (mandates) |
| **Mandate** | A PoA credential as defined in RFC 0116: the complete authorization document specifying an agent's permissions, constraints, budget, and governance context |
| **Governance Profile** | One of five permission boundary definitions (minimal, standard, strict, enterprise, behörde) as defined in RFC 0115 |
| **Ceiling** | The maximum permitted value for a mandate attribute within a given governance profile |
| **Ceiling Violation** | A mandate attribute that exceeds the ceiling defined by the mandate's governance profile |
| **Registration Path** | A method by which a mandate is created and validated. This specification defines Path 4 (rule-based); Paths 1–3 are proprietary AI-assisted paths |
| **Status Transition** | A change in mandate lifecycle state (e.g., DRAFT → ACTIVE, ACTIVE → REVOKED) |
| **Suspension** | A temporary, reversible deactivation of an ACTIVE mandate. The mandate retains its identity, scope, budget, and audit trail. PEP enforcement rejects suspended mandates on CHK-02 (Temporal & Status). |
| **Administration Interface** | A visual user interface for managing mandates, delegations, budgets, and governance profiles |

## 3. ARCHITECTURAL OVERVIEW

### 3.1 Management API in the GAuth Architecture

The Management API sits between the administrator (or automated provisioning system) and the mandate store. It governs how mandates enter and exit the system, while the PEP (RFC 0117) governs how they are enforced at runtime.

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Administrator / Provisioning System                                    │
│                                                                         │
│  1. Submit mandate creation request                                     │
│  2. Provide all required parameters (scope, budget, governance profile) │
└───────────────────────┬─────────────────────────────────────────────────┘
                        │ Management Request
                        ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  MANAGEMENT API — RFC 0118                                              │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │  1. Schema Validation (RFC 0116 §4.3)                            │  │
│  │  2. Ceiling Enforcement (RFC 0115 governance profile ceilings)   │  │
│  │  3. Consistency Checks (6 deterministic rules)                   │  │
│  │  4. Scope Checksum Computation                                   │  │
│  │  5. Audit Record Generation                                      │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  Result: ACCEPTED (mandate stored) or REJECTED (structured errors)      │
└───────────────────────┬─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  MANDATE STORE                                                          │
│                                                                         │
│  Persisted mandate available for:                                       │
│  • PEP enforcement (RFC 0117) — stateful mode reads live mandate state  │
│  • PEP budget reporting — consumption deductions written back to store  │
│  • Token issuance (RFC 0116) — OAuth engine embeds PoA in JWT/VC       │
│  • Administration Interface — query, inspect, modify                    │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 Mandate Lifecycle States

Mandates follow a defined lifecycle with deterministic state transitions:

```
                    ┌──────────┐
                    │  DRAFT   │
                    └────┬─────┘
                         │ activate()
                         ▼
                    ┌──────────┐
              ┌─────│  ACTIVE  │─────┬──────────┐
              │     └────┬─────┘     │          │
              │          │           │   suspend()
     revoke() │  expire()│  exhaust()│          │
              │          │  budget   │          ▼
              │          │           │   ┌───────────┐
              │          │           │   │ SUSPENDED │
              │          │           │   └─────┬─────┘
              │          │           │         │ resume()
              │          │           │         │ (→ ACTIVE)
              ▼          ▼           ▼         │
         ┌─────────┐ ┌─────────┐ ┌──────────────┐ ┌────────────┐
         │ REVOKED │ │ EXPIRED │ │BUDGET_EXCEEDED│ │ SUPERSEDED │
         └─────────┘ └─────────┘ └──────────────┘ └────────────┘
```

| State | Description | Terminal |
|-------|-------------|----------|
| `DRAFT` | Mandate created but not yet active. May be modified. Not enforceable by PEP. | No |
| `ACTIVE` | Mandate is live and enforceable. Budget is being consumed. Scope is immutable. | No |
| `SUSPENDED` | Mandate temporarily deactivated. PEP rejects enforcement requests (CHK-02). Scope, budget, and audit trail are preserved. May be resumed to ACTIVE or revoked to terminal. TTL clock continues during suspension. | No |
| `EXPIRED` | TTL has elapsed. Automatically transitioned by the system. | Yes |
| `REVOKED` | Explicitly revoked by an authorized party. | Yes |
| `BUDGET_EXCEEDED` | Budget fully consumed. Automatically transitioned by the system. | Yes |
| `SUPERSEDED` | Replaced by a newer mandate for the same agent + project. | Yes |

**Transition rules:**
- `DRAFT → ACTIVE`: Requires successful validation (schema + ceiling + consistency). Irreversible once scope is committed.
- `ACTIVE → SUSPENDED`: Requires caller identity with suspension authority (issuer or approver). Immediate effect — PEP enforcement rejects on next check.
- `SUSPENDED → ACTIVE`: Requires caller identity with resumption authority (issuer or approver). Mandate resumes enforcement. TTL is not extended by suspension duration.
- `SUSPENDED → REVOKED`: A suspended mandate may be revoked directly without resuming first.
- `ACTIVE → REVOKED`: Requires caller identity with revocation authority. Immediate effect — PEP enforcement rejects on next check.
- `ACTIVE → EXPIRED`: System-initiated when `ttl_seconds` elapses from activation time. Also applies to SUSPENDED mandates — a suspended mandate whose TTL elapses transitions directly to EXPIRED.
- `ACTIVE → BUDGET_EXCEEDED`: System-initiated when `remaining_cents` reaches zero.
- `ACTIVE → SUPERSEDED`: System-initiated when a new mandate is activated for the same `(agent_id, project_id)` pair. This transition MUST be atomic — see §6.4.
- Terminal states are irreversible. No transition FROM a terminal state is permitted.

**Suspension rationale:** Suspension enables temporary deactivation during security investigations, compliance reviews, or administrative holds without destroying the mandate's identity, scope, budget state, or audit trail. The alternative — revoke and re-create — loses continuity and requires re-validation.

### 3.3 Mutability Rules

| Mandate State | Scope | Budget Ceiling | TTL | Status | Metadata |
|---------------|-------|----------------|-----|--------|----------|
| `DRAFT` | Mutable | Mutable | Mutable | Mutable (→ ACTIVE only) | Mutable |
| `ACTIVE` | **Immutable** | Additive-only (ceiling may increase, never decrease) | Additive-only (TTL may be extended, never shortened) | Mutable (→ SUSPENDED, terminal states) | Read-only |
| `SUSPENDED` | **Immutable** | Additive-only | Additive-only | Mutable (→ ACTIVE, REVOKED, EXPIRED) | Read-only |
| Terminal | Immutable | Frozen | Frozen | Frozen | Read-only |

**Scope immutability rationale:** Once a mandate is active, its scope (governance_profile, core_verbs, platform_permissions, allowed_paths, denied_paths, phase, active_modules) is frozen. This ensures that the PEP's enforcement decisions are stable — a mandate cannot be weakened while in use. Budget ceiling and TTL are the exceptions: they may be increased (additive-only) to extend a mandate's operational life without changing its permissions.

## 4. MANAGEMENT API — MANDATE LIFECYCLE

### 4.1 Create Mandate

Creates a new mandate in `DRAFT` state.

**Method signature (language-agnostic):**

```
createMandate(request: MandateCreationRequest) → MandateCreationResponse
```

**MandateCreationRequest schema:**

```json

          "type": "string",

        "budget": {
          "type": "object",

```

**Behavior:**
1. Validate request through the three-stage pipeline (Section 10)
2. Compute `scope_checksum` (SHA-256 of canonical JSON serialization of the `scope` object)
3. Compute `tool_permissions_hash` (SHA-256 of canonical JSON serialization of `core_verbs`)
4. Compute `platform_permissions_hash` (SHA-256 of canonical JSON serialization of `platform_permissions`)
5. Generate mandate ID (UUID v4 or equivalent — MUST be unpredictable, no sequential IDs)
6. Store mandate in `DRAFT` state
7. Generate audit record


### 4.2 Activate Mandate

Transitions a `DRAFT` mandate to `ACTIVE`. Re-runs the full validation pipeline (Section 10) to ensure the mandate is still valid at activation time.

**Method signature:**

```
activateMandate(request: MandateActivationRequest) → MandateActivationResponse
```

**MandateActivationRequest:**

```json
{
  "mandate_id": "mdt_abc123",
  "activated_by": "user_789"
}
```

**Behavior:**
1. Verify mandate is in `DRAFT` state (reject if not)
2. Re-run three-stage validation pipeline (Section 10) — mandate parameters may have become invalid since creation (e.g., governance profile ceilings updated)
3. Check for existing `ACTIVE` mandate for the same `(agent_id, project_id)` pair
4. If existing active mandate found: atomically supersede it (§6.4) before activating this one
5. Set `activated_at` timestamp and compute `expires_at` = `activated_at` + `ttl_seconds`
6. Transition status to `ACTIVE`
7. Generate audit record

**MandateActivationResponse:**

```json
{
  "mandate_id": "mdt_abc123",
  "status": "ACTIVE",
  "activated_at": "2026-04-08T10:05:00Z",
  "expires_at": "2026-04-08T22:05:00Z",
  "superseded_mandate_id": null,
  "audit": {
    "operation": "ACTIVATE",
    "performed_by": "user_789",
    "timestamp": "2026-04-08T10:05:00Z",
    "mandate_id": "mdt_abc123"
  }
}
```

### 4.3 Revoke Mandate

Transitions an `ACTIVE` or `SUSPENDED` mandate to `REVOKED`. Immediate effect — the PEP will reject enforcement requests on the next check (CHK-02 temporal & status, per RFC 0117 §9.2).

**Method signature:**

```
revokeMandate(request: MandateRevocationRequest) → MandateRevocationResponse
```

**MandateRevocationRequest:**

```json
{
  "mandate_id": "mdt_abc123",
  "revoked_by": "user_789",
  "reason": "Security incident — agent compromised"
}
```

**Behavior:**
1. Verify mandate is in `ACTIVE` or `SUSPENDED` state (reject if terminal)
2. Verify caller has revocation authority (must be `issued_by` or a member of `approval_chain`)
3. Transition status to `REVOKED`
4. Cascade revocation to all child mandates in the delegation chain (§8.2)
5. Generate audit record with reason

**MandateRevocationResponse:**

```json
{
  "mandate_id": "mdt_abc123",
  "status": "REVOKED",
  "revoked_at": "2026-04-08T11:00:00Z",

```

1. Verify mandate is in `ACTIVE` state (reject if not)
2. Verify caller has suspension authority (must be `issued_by` or a member of `approval_chain`)

4. Cascade suspension to all child mandates in the delegation chain

**TTL behavior during suspension:** The TTL clock continues during suspension. A mandate that expires while suspended transitions directly to `EXPIRED`. Suspension does not extend the mandate's lifetime — use `extendTTL()` (§7.4) if additional time is needed after resumption.

## 6. MANAGEMENT API — STATUS TRANSITIONS

### 6.1 Transition Rules

Status transitions are governed by the following state machine. The Management API MUST enforce these rules; any invalid transition MUST return an error.

| From State | To State | Trigger | Authority Required |
|------------|----------|---------|-------------------|
| `DRAFT` | `ACTIVE` | `activateMandate()` | Mandate creator or approver |
| `DRAFT` | (deleted) | `deleteMandate()` | Mandate creator |
| `ACTIVE` | `SUSPENDED` | `suspendMandate()` | Issuer, approver, or system |
| `ACTIVE` | `REVOKED` | `revokeMandate()` | Issuer, approver, or delegated revocation authority |
| `ACTIVE` | `EXPIRED` | System clock (TTL elapsed) | Automatic |
| `ACTIVE` | `BUDGET_EXCEEDED` | System (remaining_cents = 0) | Automatic |
| `ACTIVE` | `SUPERSEDED` | New mandate activated for same (agent, project) | Automatic |
| `SUSPENDED` | `ACTIVE` | `resumeMandate()` | Issuer or approver |
| `SUSPENDED` | `REVOKED` | `revokeMandate()` | Issuer, approver, or delegated revocation authority |
| `SUSPENDED` | `EXPIRED` | System clock (TTL elapsed during suspension) | Automatic |

### 6.2 Draft Deletion

DRAFT mandates that have not been activated may be deleted. This is a hard delete — the mandate is removed from the store. An audit record is retained.

**Method signature:**

```
deleteMandate(mandate_id: string, deleted_by: string) → DeletionConfirmation
```

**Constraints:**
- Only `DRAFT` mandates may be deleted
- Only the mandate creator (`issued_by`) may delete
- Audit record is retained even after deletion

### 6.3 Automatic Transitions

The Management API implementation MUST handle automatic transitions:

1. **Expiration:** A background process or lazy check MUST transition `ACTIVE` and `SUSPENDED` mandates to `EXPIRED` when `activated_at + ttl_seconds < current_time`. PEP implementations performing stateful enforcement (RFC 0117 §10.3) MUST also check temporal validity and treat expired mandates as `EXPIRED` even if the store has not yet transitioned the status.

2. **Budget exhaustion:** When a budget deduction reduces `remaining_cents` to zero, the mandate MUST transition to `BUDGET_EXCEEDED`. PEP implementations checking budget (CHK-13) MUST also enforce this boundary.

3. **Supersession:** When `activateMandate()` detects an existing `ACTIVE` mandate for the same `(agent_id, project_id)` pair, the prior mandate MUST transition to `SUPERSEDED` before the new mandate is activated. See §6.4 for atomicity requirements.

### 6.4 Supersession Atomicity

The supersession transition (ACTIVE → SUPERSEDED + new DRAFT → ACTIVE) MUST be atomic. Implementations MUST ensure that at no point are two mandates simultaneously in `ACTIVE` state for the same `(agent_id, project_id)` pair.

**Implementation requirements:**
- The supersession and activation MUST be performed within a single transaction or under an exclusive lock on the `(agent_id, project_id)` pair
- If the transaction fails, neither the supersession nor the activation occurs (rollback)
- Concurrent `activateMandate()` calls for the same `(agent_id, project_id)` pair MUST be serialized — the second caller receives `DUPLICATE_MANDATE` if both target the same pair

**Rationale:** Without atomicity, a race condition between two concurrent activations could leave both mandates active, or supersede the wrong mandate. The PEP (RFC 0117 CHK-02) expects exactly zero or one active mandate per `(agent_id, project_id)` pair in stateful mode.

## 7. MANAGEMENT API — BUDGET OPERATIONS

### 7.1 Budget Model

Each mandate has a budget defined in cents:

| Field | Type | Mutability | Description |
|-------|------|-----------|-------------|
| `total_cents` | integer | Additive-only (ACTIVE/SUSPENDED); mutable (DRAFT) | Budget ceiling — may only increase once active |
| `remaining_cents` | integer | Decremented by PEP enforcement and budget consumption reports | Current balance |
| `consumed_cents` | integer (computed) | Read-only | `total_cents - remaining_cents` |

### 7.2 Increase Budget Ceiling

Increases the budget ceiling of an `ACTIVE` or `SUSPENDED` mandate. The budget ceiling may only increase, never decrease (additive-only rule).

**Method signature:**


**Behavior:**
1. Verify mandate is `ACTIVE` or `SUSPENDED`
2. Verify `additional_cents` > 0
3. Add `additional_cents` to `total_cents` and `remaining_cents` atomically
4. Generate audit record
5. Return updated budget state

**BudgetIncreaseResponse:**

```json
{
  "mandate_id": "mdt_abc123",
```

### 7.3 Report Budget Consumption

PEP implementations and enforcement systems report budget consumption back to the mandate store through the Management API. This is the normative interface for keeping budget state synchronized between the enforcement layer and the management layer.

**Method signature:**

```
reportConsumption(request: ConsumptionReport) → ConsumptionAcknowledgement
```

**ConsumptionReport:**

```json
{
  "mandate_id": "mdt_abc123",
  "deducted_cents": 42,
  "enforcement_request_id": "enf_xyz789",
  "operation_type": "agent_command_authorization",
  "timestamp": "2026-04-08T12:30:00Z",
  "reported_by": "pep:enforcement"
}
```

**Behavior:**
1. Verify mandate is `ACTIVE` (reject consumption reports for non-active mandates)
2. Verify `deducted_cents` > 0
3. Atomically decrement `remaining_cents` by `deducted_cents`
4. If `remaining_cents` reaches zero: transition mandate to `BUDGET_EXCEEDED` (§6.3)
5. Generate audit record of type `BUDGET_CONSUMPTION`
6. Return updated budget state

**ConsumptionAcknowledgement:**

```json
{
  "mandate_id": "mdt_abc123",
  "accepted": true,
  "budget": {
    "total_cents": 15000,
    "remaining_cents": 9158,
    "consumed_cents": 5842
  },
  "budget_exceeded": false
}
```

**Idempotency:** Consumption reports MUST include an `enforcement_request_id`. The Management API MUST reject duplicate reports with the same `enforcement_request_id` to prevent double-deduction. Duplicate reports return `accepted: false` with the current budget state.

**Atomicity:** Budget deduction MUST be atomic. Concurrent consumption reports for the same mandate MUST be serialized to prevent `remaining_cents` from going negative.

## 8. MANAGEMENT API — DELEGATION LIFECYCLE

### 8.1 Create Delegation

Creates a delegated mandate from an existing parent mandate. The delegated mandate's scope MUST be equal to or narrower than the parent's scope (scope narrowing rule, per RFC 0117 §11).

**Method signature:**

```
createDelegation(request: DelegationRequest) → DelegationResponse
```

**DelegationRequest:**

```json
{
  "parent_mandate_id": "mdt_abc123",
  "delegate": {
    "agent_id": "agent_sub_789",
    "description": "Sub-agent for testing tasks"
  },
  "scope_restriction": {
    "allowed_paths": ["tests/"],
    "denied_paths": ["tests/fixtures/secrets/"],
    "core_verbs": {
      "file.read": { "allowed": true },
      "file.write": { "allowed": true, "constraints": { "path_patterns": ["tests/**"] } },
      "command.run": { "allowed": false }
    }
  },
  "requirements_restriction": {
    "budget": { "total_cents": 2000 },
    "ttl_seconds": 3600,
    "max_tool_calls": 100
  },
  "delegated_by": "agent_456"
}
```

**Validation rules:**
1. Parent mandate MUST be `ACTIVE`
2. Delegator (`delegated_by`) MUST be the parent mandate's `subject` or a member of its `approval_chain`
3. Delegation depth MUST NOT exceed `max_delegation_depth` defined in the parent's verb constraints
4. Every field in `scope_restriction` MUST be equal to or narrower than the parent's scope:
   - `allowed_paths` must be a subset of or equal to parent's `allowed_paths`
   - `denied_paths` may add entries but MUST NOT remove parent's `denied_paths`
   - `core_verbs` may only disable verbs or add constraints — MUST NOT enable a verb the parent disables
   - Platform permissions may only restrict, never expand
5. Budget MUST NOT exceed parent's `remaining_cents`
6. TTL MUST NOT exceed parent's remaining TTL
7. Governance profile MUST be equal to or stricter than parent's

**Budget deduction:** When a delegation is created, the delegated budget (`total_cents`) MUST be atomically deducted from the parent mandate's `remaining_cents`. This is a reservation — the parent's available budget is reduced by the amount allocated to the child. If the parent is later revoked, any unspent child budget is not returned to the parent (parent is terminal).

```json
      "delegate": "agent_sub_789",

      "max_depth_remaining": 0
    }

  "effective_scope": { },

    "total_cents": 10000,
    "remaining_cents": 2200,
    "consumed_cents": 5800,
    "reserved_for_delegations_cents": 2000
  },
  "audit": {
    "operation": "DELEGATE",
    "performed_by": "agent_456",
    "timestamp": "2026-04-08T12:00:00Z",
    "mandate_id": "mdt_child_001",
    "parent_mandate_id": "mdt_abc123"
  }
}
```

### 8.2 Revoke Delegation

Revoking a parent mandate MUST cascade to all child mandates in the delegation chain. This is a hard requirement — no orphaned active delegations may exist.

**Cascade behavior:**
1. When a mandate is revoked, all child mandates (direct and transitive) MUST be revoked
2. Each child receives its own audit record with `reason: "PARENT_REVOKED"`
3. Cascade is synchronous — the revocation response is not returned until all children are revoked
4. Cascade also applies to suspension: suspending a parent MUST cascade suspension to all children

## 9. MANAGEMENT API — GOVERNANCE PROFILE ASSIGNMENT

### 9.1 Profile Assignment

Every mandate MUST declare a governance profile. The Management API validates the mandate's parameters against the ceiling table for the declared profile.

**Supported profiles (per RFC 0115):**

| Profile | Description | Registration Context |
|---------|-------------|---------------------|
| `minimal` | Least restrictive. Maximum autonomy. | Prototyping, personal projects |
| `standard` | Balanced restrictions. Supervised approval. | Small teams, general development |
| `strict` | Restrictive. Audited packages, allowlist shell. | Regulated industries |
| `enterprise` | Highly restrictive. No delegation, no secrets access. | Enterprise organizations |
| `behoerde` | Most restrictive. Four-eyes approval, 30-minute sessions. | Public sector / government |

### 9.2 Profile Ceiling Validation

When a mandate is created or activated, the Management API MUST validate every attribute against the governance profile ceiling:

| Attribute | Minimal | Standard | Strict | Enterprise | Behörde |
|-----------|---------|----------|--------|------------|---------|
| Deployment targets | dev, staging, prod | dev, staging | staging | staging | staging |
| Auto-deploy | ✓ | ✗ | ✗ | ✗ | ✗ |
| DB write access | ✓ | ✓ | ✓ | ✗ | ✗ |
| DB migration | ✓ | ✗ | ✗ | ✗ | ✗ |
| DB production access | ✓ | ✗ | ✗ | ✗ | ✗ |
| Shell mode | any | denylist | allowlist | allowlist | allowlist |
| Packages audited only | ✗ | ✗ | ✓ | ✓ | ✓ |
| Secrets read | ✓ | ✓ | ✓ | ✗ | ✗ |
| Secrets create | ✓ | ✗ | ✗ | ✗ | ✗ |
| Agent delegation | ✓ | ✓ (depth 1) | ✓ (depth 1) | ✗ | ✗ |
| Min. approval mode | autonomous | supervised | supervised | supervised | four-eyes |
| Max. session (min) | ∞ | 240 | 120 | 60 | 30 |
| Max. tool calls | ∞ | 500 | 200 | 100 | 100 |
| Max. lines/commit | ∞ | 500 | 200 | 100 | 100 |

**Validation behavior:**
- Any attribute exceeding its profile ceiling produces a `CEILING_VIOLATION` error
- Multiple ceiling violations are collected and returned together (not fail-fast)
- The mandate is rejected; partial acceptance is not permitted
- Ceiling values are immutable per profile — the Management API does not allow ceiling override (proprietary extensions may offer this as an AI-assisted feature)

### 9.3 Profile Upgrade / Downgrade

A mandate's governance profile MUST NOT be changed after activation (scope immutability). Profile changes require a new mandate:

1. Create a new mandate with the desired profile
2. Activate the new mandate (automatically supersedes the old one)
3. The old mandate transitions to `SUPERSEDED`

In `DRAFT` state, the governance profile may be changed freely.

## 10. VALIDATION SEMANTICS

### 10.1 Three-Stage Validation Pipeline

Every mandate creation and activation passes through a three-stage validation pipeline. All stages are deterministic and rule-based.

```
Stage 1: Schema Validation
    │ Validate against RFC 0116 §4.3 PoA Credential Schema
    │ Result: PASS or SCHEMA_ERROR[]
    ▼
Stage 2: Ceiling Enforcement
    │ Validate each attribute against governance profile ceiling (§9.2)
    │ Result: PASS or CEILING_VIOLATION[]
    ▼
Stage 3: Consistency Checks
    │ Apply 6 deterministic consistency rules
    │ Result: PASS or CONSISTENCY_ERROR[]
    ▼
Overall: ACCEPTED (all three stages pass) or REJECTED (structured errors from all stages)
```

### 10.2 Stage 1 — Schema Validation

The mandate's `scope` and `requirements` objects are validated against the canonical PoA JSON Schema (RFC 0116 §4.3, `https://gimelfoundation.com/schemas/poa/v2.2/poa-credential.json`).

**Checks:**
- All required fields present
- Field types match schema definitions
- Enum values are valid members
- `core_verbs` entries conform to the `ToolPolicy` schema
- `platform_permissions` conform to the `PlatformPermissions` schema
- `governance_profile` is a recognized profile name (reject with `PROFILE_NOT_FOUND` if unknown)

### 10.3 Stage 2 — Ceiling Enforcement

Each of the 14 ceiling attributes (7 stateless + 7 stateful) is compared against the governance profile ceiling table (§9.2).

**Checks:**
- Session duration ≤ profile maximum
- Tool call limit ≤ profile maximum
- Lines per commit ≤ profile maximum
- Approval mode meets profile minimum (e.g., `behoerde` requires `four-eyes`)
- Deployment targets are subset of profile allowed targets
- Platform permissions do not exceed profile allowances
- Delegation depth ≤ profile maximum

### 10.4 Stage 3 — Consistency Checks

Six deterministic consistency rules are applied:

| Rule | Check | Error Code |
|------|-------|-----------|
| C-1 | `four-eyes` approval mode requires `approval_chain` with ≥ 2 distinct entries | `FOUR_EYES_MISSING_APPROVERS` |
| C-2 | `denied_paths` and `allowed_paths` must not contain the same entry | `PATH_CONFLICT` |
| C-3 | Budget `total_cents` must be ≥ 0 (zero-budget mandates are valid for read-only agents) | `INVALID_BUDGET` |
| C-4 | TTL must be ≥ 60 seconds | `TTL_TOO_SHORT` |
| C-5 | If `active_modules` references a module, that module must exist in the governance profile's module registry | `UNKNOWN_MODULE` |
| C-6 | If `delegation_chain` is present (for delegated mandates), every delegation entry must satisfy scope narrowing | `DELEGATION_SCOPE_WIDENING` |

### 10.5 Validation Response

When validation fails, the Management API returns a structured error listing all violations across all three stages:

```json
{
  "accepted": false,
  "validation": {
    "schema_errors": [
      {
        "path": "$.scope.core_verbs.file.modify.allowed",
        "error": "Expected boolean, got string",
        "code": "SCHEMA_TYPE_MISMATCH"
      }
    ],
    "ceiling_violations": [
      {
        "attribute": "max_session_duration_minutes",
        "requested": 480,
        "ceiling": 240,
        "profile": "standard",
        "code": "CEILING_VIOLATION"
      }
    ],
    "consistency_errors": [
      {
        "rule": "C-1",
        "message": "four-eyes approval mode requires approval_chain with >= 2 distinct entries",
        "code": "FOUR_EYES_MISSING_APPROVERS"
      }
```

### 11.2 Authentication

All Management API endpoints MUST require authentication. The specification does not mandate a specific authentication mechanism — implementations may use:

- OAuth 2.1 bearer tokens
- Mutual TLS (mTLS)
- API keys with HMAC signatures
- Platform-specific S2S authentication (e.g., Platform Key + HMAC as used by GAuth internally)

The caller's identity MUST be captured in audit records.

**Budget consumption endpoint authentication:** The `/budget/consume` endpoint is called by PEP enforcement systems. Implementations SHOULD use a distinct authentication scope or credential for PEP-to-Management-API calls to distinguish enforcement-initiated budget deductions from administrative operations in audit trails.

### 11.3 Request/Response Format

- Content-Type: `application/json`
- Character encoding: UTF-8
- Timestamps: ISO 8601 with UTC timezone (`2026-04-08T10:00:00Z`)
- Pagination: Cursor-based (opaque cursor string)
- Error responses: Use the error schema defined in Section 12

### 11.4 Rate Limiting

Management API implementations SHOULD implement rate limiting to prevent abuse. Recommended limits:

| Operation Category | Recommended Limit |
|-------------------|-------------------|
| Create / Activate / Revoke / Suspend / Resume | 100 requests per minute per caller |
| Query / List / Get | 1000 requests per minute per caller |
| Budget increase / TTL extension | 50 requests per minute per caller |
| Budget consumption reports | 10000 requests per minute per PEP (high-throughput enforcement) |
| Delegation creation | 50 requests per minute per caller |

Rate limit responses use HTTP 429 with the error schema (Section 12).

```json
  "mgmt_version": "1.1.0",

    "suspension": true,
    "ttl_extension": true,
    "budget_consumption_reporting": true
  }
}
```

### 13.2 Compatibility Rules

- **Minor version increments** (1.0 → 1.1): Backward-compatible additions (new optional fields, new endpoints). Existing clients continue to work.
- **Major version increments** (v1 → v2): Breaking changes. New URL path prefix (`/gauth/mgmt/v2/`). Old version remains available during deprecation period.
- **Schema version dependency:** The Management API validates against a specific version of the RFC 0116 PoA schema. The `supported_schema_version` field in the health response indicates which schema version is expected.

## 14. ADMINISTRATION INTERFACE PATTERNS (NON-NORMATIVE)

This section provides non-normative guidance for building administration UIs on top of the Management API. These are functional requirements, not visual prescriptions — implementations should adapt layouts and styling to their target audience.

### 14.1 Mandate Creation Flow

A mandate creation UI SHOULD implement a multi-step flow with ceiling-aware validation:

1. **Agent & Project Selection** — Select or search for the target agent and project. Auto-fill issuer from session.
2. **Governance Profile Selection** — Select profile; display ceiling table preview on selection; disable options that would violate the selected profile's ceilings.
3. **Scope Configuration** — Phase selection, allowed/denied paths, module selection, core verb configuration with per-verb toggles and constraints, platform permission toggles.
4. **Requirements** — Approval mode, budget, TTL, session limits. All fields constrained by the selected profile's ceilings.
5. **Review & Create** — Full mandate preview (structured and/or JSON), validation status indicator, choice to create as DRAFT or create and activate immediately.

**Key UX principles:**
- Profile selection SHOULD constrain available options in subsequent steps (ceiling-aware form)
- Real-time validation feedback as the user fills in fields
- Ceiling violations shown inline (not only on submission)

### 14.2 Mandate Inspection View

A mandate inspection UI SHOULD include:

| Component | Content |
|-----------|---------|
| **Header** | Mandate ID, status badge (including SUSPENDED), agent name, project name |
| **Scope panel** | Governance profile, phase, paths, modules, verbs (expandable), platform permissions |
| **Requirements panel** | Approval mode, budget utilization, TTL remaining, session limits |
| **Delegation panel** | Delegation chain visualization, effective scope after narrowing |
| **History panel** | Chronological audit trail (create, activate, suspend, resume, budget changes, delegation, revocation) |
| **Actions** | Suspend/Resume toggle, Revoke (with confirmation), Budget increase, TTL extension, Delegate |

### 14.3 Mandate List View

A mandate list UI SHOULD display: Mandate ID, Agent, Project, Governance Profile, Status (color-coded, including SUSPENDED), Budget utilization, TTL remaining, and Creation date. Support filtering by status (including SUSPENDED), profile, agent, project, and date range.

### 14.4 Budget Monitoring

A budget monitoring UI SHOULD provide: per-mandate budget utilization display, consumption history, alerts for mandates approaching budget exhaustion (>80% consumed), and a view of top-consuming agents.

### 14.5 Delegation Management

A delegation management UI SHOULD provide: delegation chain visualization (tree or hierarchy), side-by-side scope comparison (parent vs. delegated scope highlighting restrictions), cascade warning for revocation/suspension, and a delegation creation form pre-populated with parent scope as maximum bounds.

### 14.6 Governance Profile Browser

A governance profile UI SHOULD provide: all 5 profiles displayed with ceiling values for comparison, and an impact preview showing which current DRAFT settings would violate a new profile's ceilings.

## 15. CONFORMANCE REQUIREMENTS

### 15.1 Conformant Management API

A conformant Management API implementation MUST:

1. Implement all endpoints defined in Section 11.1
2. Validate all mandates through the three-stage pipeline (Section 10.1)
3. Enforce all governance profile ceilings (Section 9.2) without exception or override
4. Enforce all status transition rules (Section 6.1) — no invalid transitions
5. Enforce scope immutability for ACTIVE and SUSPENDED mandates (Section 3.3)
6. Enforce additive-only budget ceiling and TTL changes (Sections 7.2, 7.4)
7. Enforce delegation scope narrowing (Section 8.1)
8. Enforce delegation budget deduction atomically from parent mandate (Section 8.1)
9. Enforce supersession atomicity (Section 6.4)
10. Accept and process budget consumption reports with idempotency (Section 7.3)
11. Generate audit records for all management operations
12. Return structured errors conforming to Section 12
13. Report interface version and feature support via the health endpoint (Section 13.1)

### 15.2 Conformant Administration Interface

A conformant Administration Interface implementation SHOULD:

1. Use the Management API (Section 11) as its sole data layer — no direct mandate store access
2. Display ceiling constraints during mandate creation (ceiling-aware forms)
3. Show real-time validation feedback
4. Display delegation chains with scope comparison
5. Provide budget monitoring with utilization visualization
6. Enforce confirmation dialogs for destructive operations (revocation, deletion)
7. Display SUSPENDED state distinctly from ACTIVE and terminal states

Note: Administration Interface conformance is SHOULD (non-normative), not MUST.

## 16. SECURITY CONSIDERATIONS

### 16.1 Authentication and Authorization

- All Management API endpoints MUST require authentication (Section 11.2)
- Mandate creation authority SHOULD be restricted to designated administrators or provisioning systems
- Revocation and suspension authority MUST be limited to the original issuer (`issued_by`) or members of the `approval_chain`
- Delegation authority MUST be limited to the current mandate holder (the agent) or the mandate issuer
- Budget increase and TTL extension authority SHOULD be restricted to the mandate issuer or organizational administrators
- Budget consumption reporting authority SHOULD be restricted to authenticated PEP systems

### 16.2 Audit Trail Integrity

- Audit records MUST be append-only — no modification or deletion of audit entries
- Audit records MUST include the identity of the caller, the operation performed, the timestamp, and the affected mandate
- Budget consumption audit records MUST include the `enforcement_request_id` for traceability to PEP decisions
- For regulatory environments (particularly `behoerde` profile), audit retention SHOULD comply with applicable data retention requirements

### 16.3 Fail-Closed Validation

- If the Management API cannot complete validation (schema resolution failure, ceiling data unavailable, internal error), the mandate MUST be rejected
- There is no auto-approve mode, no bypass mechanism, and no administrative override for validation failures
- This is consistent with the fail-closed principle in RFC 0117 §1.2

### 16.4 Scope Immutability Enforcement

- Once a mandate transitions from DRAFT to ACTIVE, scope MUST be immutable at the application layer
- Scope MUST remain immutable during SUSPENDED state
- Implementations SHOULD also enforce immutability at the data store layer (e.g., database triggers, immutable columns)
- Any attempt to modify scope on an ACTIVE or SUSPENDED mandate MUST return `MANDATE_NOT_DRAFT` error

### 16.5 Delegation Chain Security

- Scope narrowing MUST be enforced — a delegation MUST NOT expand the parent's scope
- Budget allocation from parent to child MUST be deducted from the parent's `remaining_cents` atomically at delegation creation time — no double-spending
- Delegation depth limits MUST be enforced at creation time, not only at enforcement time
- Cascade revocation MUST be synchronous — no window where a child mandate is active after its parent is revoked
- Cascade suspension MUST be synchronous — no window where a child mandate is active after its parent is suspended

### 16.6 Rate Limiting

- Management API implementations SHOULD implement rate limiting (Section 11.4)
- Rate limiting prevents abuse of the validation pipeline (schema validation + ceiling enforcement can be computationally expensive)
- The budget consumption endpoint SHOULD have a higher rate limit than administrative endpoints to accommodate high-throughput PEP enforcement
- Rate limit responses MUST use the standard error schema with `RATE_LIMITED` code

### 16.7 Input Validation

- All string inputs MUST be validated for maximum length (implementation-defined limits)
- Path patterns (`allowed_paths`, `denied_paths`) MUST be validated for glob syntax safety — no path traversal attacks
- Mandate IDs MUST be unpredictable (UUID v4, ULID, or equivalent) — no sequential IDs
- `enforcement_request_id` values in consumption reports MUST be validated for uniqueness (idempotency key)

### 16.8 Supersession Safety

- The atomicity requirements in §6.4 MUST be enforced to prevent the "dual active mandate" race condition
- Implementations MUST use database-level locking or compare-and-swap to guarantee that at most one mandate is ACTIVE for any given `(agent_id, project_id)` pair at any time

## Appendix A: Reference Implementation Pseudocode

### A.1 Create Mandate

```
function createMandate(request):
    // Stage 1: Schema validation
    schema_errors = validateAgainstSchema(request, POA_CREDENTIAL_SCHEMA)
    if schema_errors:
        return REJECTED(schema_errors)

    // Check governance profile exists
    profile = request.scope.governance_profile
    if not profileExists(profile):
        return ERROR(PROFILE_NOT_FOUND, profile)

    // Stage 2: Ceiling enforcement
    ceilings = getCeilingTable(profile)
    ceiling_violations = []

    if request.requirements.session_limits.max_session_duration_minutes > ceilings.max_session_minutes:
        ceiling_violations.append(CEILING_VIOLATION("max_session_duration_minutes", ...))
    if request.requirements.session_limits.max_tool_calls > ceilings.max_tool_calls:
        ceiling_violations.append(CEILING_VIOLATION("max_tool_calls", ...))
    if request.requirements.session_limits.max_lines_per_commit > ceilings.max_lines_per_commit:
        ceiling_violations.append(CEILING_VIOLATION("max_lines_per_commit", ...))
    if not meetsMinimumApprovalMode(request.requirements.approval_mode, ceilings.min_approval_mode):
        ceiling_violations.append(CEILING_VIOLATION("approval_mode", ...))
    // ... validate all 14 attributes

    if ceiling_violations:
        return REJECTED(ceiling_violations)

    // Stage 3: Consistency checks
    consistency_errors = []
    if request.requirements.approval_mode == "four-eyes":
        if not request.parties.approval_chain or len(request.parties.approval_chain) < 2:
            consistency_errors.append(CONSISTENCY_ERROR("C-1", "FOUR_EYES_MISSING_APPROVERS"))

    overlap = intersection(request.scope.allowed_paths, request.scope.denied_paths)
    if overlap:
        consistency_errors.append(CONSISTENCY_ERROR("C-2", "PATH_CONFLICT"))

    if request.requirements.budget.total_cents < 0:
        consistency_errors.append(CONSISTENCY_ERROR("C-3", "INVALID_BUDGET"))

    if request.requirements.ttl_seconds < 60:
        consistency_errors.append(CONSISTENCY_ERROR("C-4", "TTL_TOO_SHORT"))

    // ... remaining consistency checks

    if consistency_errors:
        return REJECTED(consistency_errors)

    // All stages passed — create mandate
    mandate = Mandate(
        id = generateMandateId(),  // UUID v4, unpredictable
        status = DRAFT,
        parties = request.parties,
        scope = request.scope,
        requirements = request.requirements,
        scope_checksum = computeScopeChecksum(request.scope),
        tool_permissions_hash = computeHash(request.scope.core_verbs),
        platform_permissions_hash = computeHash(request.scope.platform_permissions),
        created_at = now()
    )

    store(mandate)
    auditLog(CREATE, mandate.id, request.parties.issued_by)
    return ACCEPTED(mandate)
```

### A.2 Create Delegation

```
function createDelegation(request):
    parent = getMandate(request.parent_mandate_id)
    if parent.status != ACTIVE:
        return ERROR(PARENT_MANDATE_NOT_ACTIVE)

    // Verify delegation authority
    if request.delegated_by != parent.parties.subject
       and request.delegated_by not in parent.parties.approval_chain:
        return ERROR(INSUFFICIENT_AUTHORITY)

    // Check delegation depth
    current_depth = len(parent.delegation_chain)
    max_depth = getMaxDelegationDepth(parent.scope)
    if current_depth >= max_depth:
        return ERROR(DELEGATION_DEPTH_EXCEEDED)

    // Verify scope narrowing (two-pass, per RFC 0117 §11)
    if not isScopeSubset(request.scope_restriction, parent.scope):
        return ERROR(DELEGATION_SCOPE_WIDENING)

    // Verify budget
    if request.requirements_restriction.budget.total_cents > parent.requirements.budget.remaining_cents:
        return ERROR(DELEGATION_BUDGET_EXCEEDED)

    // Verify TTL
    parent_remaining_ttl = parent.expires_at - now()
    if request.requirements_restriction.ttl_seconds > parent_remaining_ttl:
        return ERROR(DELEGATION_TTL_EXCEEDED)

    // Atomically: deduct budget from parent + create child mandate
    atomicTransaction:
        deductBudget(parent, request.requirements_restriction.budget.total_cents)
        child_mandate = createMandate(buildDelegatedMandateRequest(parent, request))
        child_mandate.delegation_chain = parent.delegation_chain + [newDelegationEntry(request)]
        activateMandate(child_mandate.id)  // delegations skip DRAFT

    auditLog(DELEGATE, child_mandate.id, request.delegated_by, parent.id)
    return DelegationResponse(child_mandate)
```

### A.3 Report Consumption

```
function reportConsumption(request):
    // Idempotency check
    if consumptionAlreadyProcessed(request.enforcement_request_id):
        return ConsumptionAcknowledgement(accepted=false, budget=getCurrentBudget(request.mandate_id))

    mandate = getMandate(request.mandate_id)
    if mandate.status != ACTIVE:
        return ERROR(MANDATE_NOT_ACTIVE)

    // Atomic budget deduction
    atomicTransaction:
        mandate.requirements.budget.remaining_cents -= request.deducted_cents
        if mandate.requirements.budget.remaining_cents <= 0:
            mandate.requirements.budget.remaining_cents = 0
            mandate.status = BUDGET_EXCEEDED
            auditLog(STATUS_TRANSITION, mandate.id, "system", "BUDGET_EXCEEDED")
        recordConsumption(request.enforcement_request_id, request.deducted_cents)

    auditLog(BUDGET_CONSUMPTION, mandate.id, request.reported_by, request.enforcement_request_id)
    return ConsumptionAcknowledgement(accepted=true, budget=mandate.requirements.budget)
```

| OQ-4 | Should the Management API support mandate cloning (create a new DRAFT from an existing mandate's scope)? | Common workflow for renewal scenarios | Recommend for v1.2. Useful for TTL-based renewal workflows where the scope should be preserved. |
| OQ-6 | Should the Management API support webhook notifications for status transitions? | Enables reactive systems (e.g., alert on revocation, notify on budget exhaustion) | Recommend for v1.2. Auth already has webhook infrastructure. |
| OQ-7 | Should the Administration Interface specification include role-based access control (RBAC) patterns? | Different UI views for administrators, project leads, and auditors | The Management API itself should define who can do what (create/revoke/delegate authority) normatively. RBAC for the Admin UI is implementation-specific. |
| OQ-8 | How should the Management API handle cross-platform mandate portability? | A mandate created on Platform A may need to be recognized by Platform B | Deferred. This is RFC 0116 territory — the token format enables portability. The Management API is local to each platform. |

**Resolved in v1.1:**

| OQ-5 (Budget consumption reporting) | **Resolved.** Section 7.3 defines the `reportConsumption()` interface and the `/budget/consume` endpoint. PEP implementations report consumption to the Management API with idempotent enforcement request IDs. |
